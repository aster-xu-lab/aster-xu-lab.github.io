订单中心慢SQL治理：从日均1124条到6条的实战总结

订单中心慢SQL治理：从日均1124条到6条

背景

我们订单中心日均处理100万+订单，高峰期慢SQL日志日均达到 1124条，Doris慢查询日均 8735条。系统可用性在 99.9% 附近徘徊，每次大促都是一次心跳加速的考验。

这篇文章记录了我们如何通过系统性治理，将慢SQL从 1124 降到 6 条（↓99.4%），系统可用性提升到 99.99%。

一、先看问题：慢SQL从哪里来？

1.1 缺失索引

这是最常见的慢SQL原因。我们通过慢查询日志聚合发现，Top 10 慢SQL中有 7 条是走全表扫描：

修复：新增联合索引 30+ 处，原则是”高频查询条件+排序字段”。

1.2 大 IN 查询

渠道订单表经常出现 WHERE channel IN (1,2,3,...,20) 的场景，IN 列表过长导致 MySQL 优化器选择了错误的执行计划：

修复：大 IN 拆分为多批次查询，每批控制在 10 个以内，或改为 JOIN 方式。

1.3 未命中分区——Doris慢查询的根源

Doris 的 8735 条慢查询中，超过 80% 是因为 SQL 没有带分区字段。Doris 分区剪枝是性能的关键，不带分区键等于全表扫描所有分区。

修复：强制所有 OLAP 查询必须带上 dt（日期分区）条件，在 DAO 层增加拦截校验。

二、连接池隔离——舱壁模式的应用

问题

所有业务模块共用一个数据库连接池，一个慢查询就能拖垮整个服务——线程池耗尽，其他正常请求全部阻塞。

方案：舱壁模式（Bulkhead Pattern）

使用 Druid 多数据源配置，为核心链路（订单查询/写入）分配独立连接池；报表导出等重查询走单独的只读连接池，即使堵死也不影响核心链路。

三、事务粒度控制

我们发现了大量”长事务”——一个事务里夹杂了 RPC 调用、文件操作甚至发MQ消息：

一个事务锁持有时间 = 两次SQL + 两次RPC + 一次MQ ≈ 500ms。高并发下数据库连接迅速耗光。

修复原则：

  1. 事务内不含 RPC
  2. 事务内不含 MQ
  3. 先做所有外部调用，最后开事务写库

四、ThreadLocal 泄漏修复

在 SyncTask 引擎中，JobContext 使用 ThreadLocal 存放批次上下文。由于线程池复用，前一个任务的 ThreadLocal 没有清理干净，导致后续任务拿到了”脏”上下文，产生了诡异的慢查询——SQL走了错误的参数。

五、Doris 分区和物化视图

Doris 慢查询治理的核心就两个字：分区。

我们对所有核心表按 dt 字段做了 Range 分区，并且为高频聚合查询（渠道日报、状态汇总）创建了物化视图，把聚合计算从查询时挪到了写入时。

总结

慢 SQL 治理看似是 DBA 的活，但在业务开发中，开发人员写出来的 SQL 才是根源。几个核心原则：

  1. 索引跟着查询走——分析慢日志，找到高频查询，建立覆盖索引
  2. 连接池隔离——重查询和核心查询不能共用同一批连接
  3. 事务要短——内部不含 RPC、MQ
  4. ThreadLocal 必须清理——线程池环境下是定时炸弹
  5. Doris 分区是生命线——不带分区键的查询就是灾难

这些原则不是一次性做完的，而是在日常 Code Review 和压测中持续迭代。
