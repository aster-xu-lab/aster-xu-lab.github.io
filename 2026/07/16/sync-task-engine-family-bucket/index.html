SyncTask Engine 全家桶：一个依赖就能跑的轻量级任务编排引擎

背景

2024 年，订单中心需要把单据同步到 10+ 个外部系统。每接一个新系统就写一个 xxl-job Handler，重复代码越来越多。

一年后 WMS 中心抄了这套设计，44 个 Handler。然后我们就陷入了两个项目维护同一套编排逻辑的窘境。

2026 年 7 月，SyncTask Engine 来了。但第一版其实只是个核心引擎——接入方得自己写 Java Config 组装 Bean，引用多个依赖。不够开箱即用。

这一轮重构的思路很明确：像 Spring Boot 官方 Starter 那样，一个依赖，默认自带最佳实践。

为什么不直接用现成的

方案 类型 为什么不行
xxl-job 调度框架 上下层关系。xxl-job 是「闹钟」，引擎是「起床后的流程」
Temporal Durable Workflow 独立 Server + 调度体系，跟 xxl-job 冲突
Conductor DAG 编排 独立 Server + ES，运维扛不住
nFlow 嵌入式工作流 DB 状态机理念接近，但引入了完整 FSM 概念
Flowable BPMN 审批 领域不同——人工审批 vs 系统自动同步
DBOS Transact 持久化执行库 只支持 Postgres

五个硬约束决定了结论：

  1. 公司所有定时任务是 xxl-job → 不能引入第二套调度
  2. 团队小（传统行业 IT） → 维护不了独立引擎集群
  3. MySQL 是唯一基础设施 → Postgres / ES 方案出局
  4. 外部系统 10+ 且持续增加 → 必须可扩展
  5. 外部系统不可靠 → 必须持久化状态机

核心设计

引擎与调度分离

八个 SPI 扩展点

SPI 默认实现 说明
SchedulerAdapter xxl-job ✅ 调度框架适配
SyncTaskParamParser Jackson 反射 ✅ JSON → SyncTaskParam
TaskStore MyBatis-Plus ✅ 任务 CRUD，自带建表 DDL
LockService Redisson ✅ 分布式锁
NotifyChannel 待实现 企微/飞书/钉钉
TaskArchiveService MyBatis ✅ 历史任务归档
TaskLifecycleListener 接入方自定义 Metrics/审计/编排链
SyncTaskParamValidator 接入方自定义 参数校验规则

带 ✅ 的是 Starter 自带的默认实现，引入即用。

架构减法

不做的事情和做的事情同样重要：

  • ✅ Handler 自动注册 → ❌ DAG 编排（90% 任务无依赖，有依赖用 afterExecute）
  • ✅ 状态加权拉取 → ❌ 独立调度 Server（寄居在 xxl-job）
  • ✅ 多 Job 分工 + 差异化 cron → ❌ 内部延迟队列（组合现有基础设施）
  • ✅ sync_task 表就是任务队列 → ❌ 完整的 MQ

每一行不写的代码，都是不用维护的资产。

状态机设计

关键设计：告警不是每次都推送，通过模运算控制频率。既不会漏掉真正的故障，也不会被偶尔的网络抖动轰炸。

模块结构

设计决策记录

几个关键决策背后的思考：

1. 为什么不用 Lombok？

core 模块零框架依赖是最重要的原则。Lombok 虽然是编译期注解，但引入了 IDE 插件依赖和编译链耦合。十几行 getter/setter 不值得为一个注解牺牲纯净度。

2. 为什么日志不用常量？

你可能会在代码里看到大量 "[SyncTaskEngine] 开始执行, 参数: {}" 这种直接写的字符串，而不是 LOG_EXECUTION_START 常量。

这是故意的。Spring Boot 源码里也是直接写字符串——排查问题时搜日志原文就能定位代码，比搜常量名效率高得多。日志常量反而增加了查错成本。

3. limitRetryTimes 为什么不在表里？

最初差点把 limitRetryTimes 写入表结构。但实际业务中，不同外部系统的重试策略完全不同——聚水潭可以重试 10 次，金蝶可能只允许 3 次。落表的话，换个策略还得跑 UPDATE，不如 job param 动态控制灵活。

4. requestParam 为什么不存？

引擎不负责序列化/反序列化请求参数。这是 Handler 自己通过 SyncTaskParamParser 解析的运行时数据，存表没有意义，反而膨胀存储。

总结

这一轮重构完成了三件事：

  1. 全家桶 Starter — 仿照 Spring Boot 官方做法，一个依赖 = xxljob + mybatis + redisson + jackson 全套默认实现，exclude 即可替换
  2. 标准表结构 — sync_task + sync_task_archive，字段经过 2 年生产验证，自带 DDL
  3. 零 Java Config — SPI Bean 只需 @Component，引擎自动发现注入

不是什么惊世骇俗的架构创新，而是一个从实际痛点长出来的工具。如果它能帮你少写几次重复的 xxl-job Handler，目的就达到了。
