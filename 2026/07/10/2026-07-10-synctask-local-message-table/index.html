SyncTaskJob 就是一张本地消息表——只是你自己没意识到

背景

前几天拆解了订单中心的 SyncTaskJob 调度引擎，发现它的核心是一张 sync_task 表：

业务代码在创建订单的同时，往这张表里插入一条”待同步”的记录。然后 xxl-job 定时轮询这张表，把状态为 INIT/FAIL/WAIT 的记录捞出来，分发给对应的 Handler 推送到外部系统（聚水潭、金蝶、FMS 等）。

这个模式太眼熟了——它就是**本地消息表（Transactional Outbox）**的变体，只是当初设计的时候没有意识到这个名词。

本地消息表是什么

最早来自 eBay 2008 年的一篇论文，后来被阿里巴巴的 RocketMQ 事务消息方案推广。核心做法是：

在同一个数据库事务里，业务数据 + 一条”待发送”的消息记录一起写入。然后后台定时轮询消息表，把未发送的消息推送到外部系统或 MQ。

关键约束有四个：

  1. 同事务：业务数据和消息记录必须在一个 BEGIN/COMMIT 里
  2. 至少一次投递：轮询 → 发送 → 标记完成，崩溃重试可能导致重复发送
  3. 投递端不幂等：消息可能重复发送（发完了还没更新状态就崩了）
  4. 消费端幂等：接收方自己处理重复消息

SyncTaskJob 对照本地消息表

把 SyncTaskJob 的做法拆开看：

本地消息表要素 SyncTaskJob 对应 匹配？
消息记录表 sync_task 表 ✅
定时轮询 xxl-job cron → SyncTaskEngine ✅
状态字段 status: INIT/FAIL/WAIT/SUCCESS/CANCEL ✅
重试机制 FAIL → 下次 cron 重新拉取执行 ✅
投递端幂等 @XLock + 二次查库 + 状态判断 ✅
消费端幂等 Handler 内部实现（接口幂等、去重） ✅
同事务保证 未强制——靠调用方在 @Transactional 中调用 syncTaskService.save() ⚠️

做到了 80%。唯一的缺口是”同事务强制”——SyncTaskJob 没有在引擎层强制要求业务数据和消息记录在同一事务中写入。如果调用方不小心把 syncTaskService.save() 放到了事务外部，消息可能丢失。

但这个问题补起来极简单：在 save() 方法上加 @Transactional(propagation = MANDATORY) 或提供一个 saveOnCommit() 方法，通过 TransactionSynchronizationManager 在事务提交后才写入。

SyncTaskJob 比标准本地消息表多做了什么

标准本地消息表是一个非常薄的设计——它就是一张表 + 一个轮询线程 + 一个发送器。它不关心发给谁、发送失败后怎么办、不同类型的消息怎么区分处理。

SyncTaskJob 在这之上加了三层：

1. 策略路由

消息不是统一发送的。sync_task 表里有 orderType（单据类型）、syncTaskType（任务类型）、syncSystem（目标系统）三个维度，引擎按这三个维度自动路由到对应的 Handler。同样是”推送订单”这件事，推聚水潭的 Handler 和推金蝶的 Handler 完全不一样，但共用了同一套轮询→执行→状态管理→告警的编排框架。

2. 告警分级

不是所有失败都要告警——标准本地消息表轮询失败后日志记一下就行。SyncTaskJob 引入了三种告警策略：单条阈值、批量阈值、最小值阈值。按重试次数取模决定是否推送企微通知，避免失败量大时刷屏。

3. 权重调度

标准本地消息表把所有未发送的消息一起捞出来发送。SyncTaskJob 做了状态分层——INIT（新任务）高频处理，FAIL（失败重试）低频处理，WAIT（等待中）中频处理。线上通过同一个 Handler 配多个 xxl-job、每个用不同的 cron 和 statusWeights 来实现差异化心跳。

这个发现意味着什么

SyncTaskJob 落地了一个经典分布式模式的变体，但当初设计的时候没有用这个术语。

这不是丢人的事——正好相反，它说明设计者独立走到了和业界标准一致的方向。这比”学了一个模式然后照着做”更能体现架构直觉。

如果要在简历或面试中提到，可以这样说：“基于本地消息表模式实现了一套任务编排引擎，在标准消息表基础上扩展了策略路由、权重调度和分级告警能力。”

本文基于对 SyncTaskJob 源码的分析。本地消息表模式的更多细节见 eBay 2008 年论文及 RocketMQ 事务消息文档。
