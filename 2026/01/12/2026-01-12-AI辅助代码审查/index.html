AI辅助代码审查实战：用Claude审查5.3万行供应链代码

AI 辅助代码审查实战：用 Claude 审查 5.3 万行供应链代码

背景

供应链计划系统（SCP）是我们从兄弟团队接手的一个项目，567 个 Java 文件，5.3 万行代码。接手后心里没底——代码质量到底怎么样？有哪些隐藏的坑？

传统的 Code Review 方式面对这个量级根本不可能完成。我决定用 Claude 做一次全量代码审查。这篇文章记录完整过程和发现。

一、怎么喂代码给 AI？

5.3 万行不可能一次性全喂进去。我采用分模块 + 结构化 Prompt 的策略：

按模块分批喂入：

  • Round 1: Service 层（18 个文件）
  • Round 2: Controller 层（12 个文件）
  • Round 3: Mapper/DAO 层（25 个文件）
  • Round 4: Model/Domain 层（30+ 个文件）
  • Round 5: 工具类和配置（15 个文件）

二、发现 1：三个 God Class

AI 第一个发现的问题就是 God Class——三个 Service 类承担了过多职责：

类名 行数 方法数 职责
SupplyChainPlanServiceImpl 1832 47 补货+铺货+审批+导出+通知…
InventoryCalculateServiceImpl 1205 31 计算+校验+日志+报表…
ReplenishOrderServiceImpl 987 28 创建+修改+审批+同步+对账…

典型症状：一个类里有 calculateReplenish()（补货计算）、generateReport()（报表生成）、sendNotification()（通知发送）三种完全不同职责的方法。

修复方案：按职责拆分

三、发现 2：26 个软删除遗漏

AI 在 Mapper XML 中发现了一个模式：很多 DELETE 语句没有改成软删除（逻辑删除）：

一共找出 26 个应改未改的硬删除。这个问题靠人工 Review 几乎不可能发现——需要遍历所有 Mapper XML 的 DELETE 语句，太枯燥了。

四、发现 3：多处 N+1 查询

修复方案：批量查询

如果 N=100，就是从 301 次 SQL 降到 4 次。

五、AI 审查的局限性

AI 也不是万能的。它发现了代码层面的问题，但有两个领域做得不好：

  1. 业务逻辑正确性：AI 看不出补货公式里 安全库存 = max(日均销量×提前期, 最小陈列量) 这个公式是否合理（这是业务专家的事）
  2. 跨系统影响：AI 不知道改了 subscribePlanStatus 会影响下游的 WMS 系统（这是架构上下文的事）

所以 AI 能做的是结构性审查（God Class、N+1、软删除、命名、异常处理），业务上下文审查仍然需要人来判断。

六、经验总结

  1. 分模块喂入：不要试图一次喂所有代码，按模块分 5-8 批
  2. 结构化 Prompt：明确审查维度，AI 输出的质量取决于 Prompt 的清晰度
  3. 验证+分类：AI 可能误报，需要人来逐条验证（我们验证后确认了约 85% 是真实问题）
  4. 排优先级：God Class 和 N+1 是 P0，命名规范是 P2，分批修复

这次审查总共发现了 50+ 个有效问题，花了约 2 个工作日（含验证）。如果纯人工搞，至少 2 周。

AI 不会替代 Code Review，但可以让 Review 的效率提升 5-10 倍。
