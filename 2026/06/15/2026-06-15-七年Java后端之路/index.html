七年Java后端之路：我的技术成长与面试方法论

七年 Java 后端之路：我的技术成长与面试方法论

写在前面

2019 年 1 月入行，到 2026 年，整整七年。

从宏智科技的外包开发，到通天晓的技术主管，再到能良的技术主管带 4 人团队——回头看，每个阶段都在做对的事情，只是当时不知道。

这篇文章不是成功学，是一个普通 Java 程序员的真实成长复盘。希望能给还在路上的同学一些参考。

第一阶段：野蛮生长（2019-2021，宏智科技）

刚毕业进入宏智，做的是传统软件外包。这个阶段最大的收获是把基础打扎实了：

  • 从 Servlet 到 Spring Boot，一套一套地啃
  • 跟着宋红康的视频系统学了 JVM（17 篇博客笔记，至今受益）
  • 把 23 种设计模式用 Java 写了一遍（25 篇博客笔记）
  • 蓝桥杯省赛一等奖、国赛优秀奖——算法功底是在这个阶段建立的

第一份工作给不了你大厂光环，但给了你安安稳稳打好基础的时间。如果在宏智那两年浑浑噩噩只做 CURD，后面的路会难走很多。

这个阶段的教训：前两年别急着追新技术，把 Java 基础、JVM、并发、设计模式啃透。这些东西十年都不会过时。

第二阶段：业务复杂度爆发（2021-2024，通天晓）

从传统软件跳到电商供应链，代码量和业务复杂度直接上了一个数量级。

OFS 订单履行系统 + WMS 仓储管理系统，4000+ Java 文件，176 个 Mapper XML，日均处理百万级订单。这段经历教会我三件事：

1. 性能优化不是玄学

系统的 QPS 从 2000 优化到 8000+，不是靠”加机器”，而是靠：SQL 优化 + MQ 解耦 + 阻塞队列批处理 + 二级缓存。每次优化都有数据支撑，不是拍脑袋。

2. 分布式事务是绕不过去的坎

AT → TCC 的迁移是我第一次独立做架构决策。压测数据告诉我 AT 不行了，调研结果告诉我 TCC 能提升 40%+。做技术选型最重要的不是”哪个方案好”，而是”哪个方案适合当前阶段”。

3. 带团队比写代码难

从一个人写代码到带 4 人团队，最难的不是技术，而是分活。怎么让每个人做适合自己的事，怎么保证代码质量一致，怎么在 deadline 和代码质量之间做平衡——这些没有教程，全靠实操。

第三阶段：架构影响力（2024-至今，能良）

能良时期是我技术影响力最大的阶段。核心产出三件事：

1. SyncTask 引擎

93 个 Handler，V2 版本作者，从 CountDownLatch 升级到 CompletableFuture。做了放弃 DAG 的架构决策——评估了 Temporal、Conductor、nFlow 等 6 个方案后选择了 Choreography 模式。这是我技术判断力最自信的时刻。

2. 稳定性治理

99.9% → 99.99%，慢 SQL ↓99.4%。不是惊天动地的创新，是 30+ 索引 + 大 IN 拆分 + 连接池隔离 + ThreadLocal 修复的具体工作。

3. Spec Coding + AI 辅助

工单系统从 0 到 1——Spec→领域建模→API 设计→ADR→AI 加速编码。182 个文件、1.2 万行代码，3 周提测。证明了”先想清楚再动手”的效率远超”先写了再说”。

我的面试方法论

面了两年、被面了无数次，总结了一套方法论：

一面（技术面）：细节为王

一面官想验证的是”你到底有没有做过”。所以回答必须带细节：

  • ❌ “我们做了性能优化”
  • ✅ “我们通过大 IN 拆分和连接池隔离，把慢 SQL 从 1124 降到 6 条”

数字比形容词有说服力。

二面（设计面）：trade-off 是核心

二面官关心的是”你为什么这么做”。所以回答必须带决策过程：

  • ❌ “我们用了策略模式”
  • ✅ “我评估了 6 个编排方案后选了 Choreography 模式，因为我们的同步链只有 11 个 Handler、链结构半年不变。引入 DAG 编排引擎的投入产出比不成比例”

展示 trade-off 过程比展示技术栈重要 10 倍。

三面（价值观面）：诚实地说”我不懂”

三面官阅人无数，装懂的都会被识破。遇到没做过的东西：

  • ❌ “这个东西我了解一些…”（然后支支吾吾）
  • ✅ “这个我确实没做过。但类似的分库分表我做过 ShardingSphere-JDBC 的改造，原理是通用的——分片路由→改写→归并”

承认不懂但展示学习迁移能力，比假装懂强 100 倍。

我为面试准备的三句话

每段经历准备三个版本：

  1. 1 分钟版（电梯演讲）：一句话概括 + 最亮眼的量化成果
  2. 3 分钟版（标准回答）：背景→方案→决策→结果
  3. 深挖版：每个技术点准备好被追问 3 层的答案（第一层怎么做、第二层为什么、第三层如果重来会怎么改进）

接下来五年的方向

  1. 即时零售中台——从异步批量到事件驱动秒级响应，补上”高时效分布式系统”这块拼图
  2. 大厂跳板——两段跳策略：先二线（携程/小红书/SHEIN），1.5-2 年后一线
  3. Agent 技术储备——SyncTask 引擎的设计经验完全可以迁移到 Agent 编排场景
  4. 持续输出——技术博客保持年更 2-4 篇，口语表达每周练

写在最后

七年下来，最大的感悟：技术能力决定下限，决策能力决定上限。

P6 能把功能做出来，P7 能做出正确的技术选型并讲清楚为什么。从 P6 到 P7 的跨越，不在于多会写代码，而在于能在一个模糊的技术场景下，做出最符合当前约束的决策，并且能说服别人。

如果只分享一条建议：每次做完一个项目，花 30 分钟写一份 ADR，记录”当时为什么这么做”。这份文档三年后回头看，比代码本身更有价值。
