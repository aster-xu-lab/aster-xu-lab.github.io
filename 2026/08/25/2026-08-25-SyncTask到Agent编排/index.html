从SyncTask到Agent：一个任务编排引擎开发者的视角

从 SyncTask 到 Agent：一个任务编排引擎开发者的视角

一个意外的发现

去年开始关注 AI Agent 开发，看了 LangChain、CrewAI、AutoGen 的文档。越看越发现一件很有意思的事：

Agent 编排引擎的核心问题，和我三年前做的 SyncTask 引擎，几乎一模一样。

这说明一个道理：基础架构能力有很强的跨领域迁移能力。这篇文章聊聊我是怎么从 SyncTask 引擎的视角来理解 Agent 开发的。

一、概念对照表

SyncTask 引擎概念 Agent 对应概念 迁移难度
HandlerManager 路由 LLM 推理 → 选 Tool 低
Handler.executeCore() Tool 被调用 → 返回结果 极低
afterExecute 埋点（Choreography） Agent Planning（LLM决定下一步） 极低
指数退避 + 状态权重 Agent 自我纠错 + 重试策略 极低
分布式锁 + 二次查库 Agent 幂等 + 超时降级 极低
context.properties（上下文透传） Agent Context / Memory 低
四级告警（SKIP/ONE/BATCH/MIN） Agent 可观测性 + 告警 低
ShardingSphere 选型方法论 Agent 框架选型（LangChain/CrewAI/AutoGen） 低
LLM 推理替代规则引擎 — 中等（需要理解 LLM 能力边界）
Python 生态（Java 为主） — 中等（需要学 Python）

核心差异只有一条：Agent 把”谁来决策下一步”这件事从代码逻辑交给了 LLM。

三、Choreography → Agent Planning

我在 SyncTask 中放弃 DAG（Orchestration），选择了 Choreography：

这个模式在 Agent 世界里就是 Agent Planning：

两者都不是”中央控制器预先排好所有步骤”，而是每一步完成后由决策者（afterExecute/LLM）决定下一步做什么。

这不是巧合。Choreography 模式天然适合”不确定下一步是什么”的场景，而 Agent Planning 也是——LLM 在每一步之后根据结果动态决定下一步。

四、可靠性设计全部可以迁移

SyncTask 引擎中最让我自豪的可靠性设计，在 Agent 中完全适用：

双重防重 → Agent 幂等

指数退避 → Agent 自我纠错

四级告警 → Agent 可观测性

五、唯一的新东西：理解 LLM 的能力边界

LLM 不是规则引擎。规则引擎可以精确输出 next_handler = "INVENTORY_SYNC"，但 LLM 可能输出”看起来合理但实际上是错的”的下一步。

所以 Agent 设计需要约束 LLM 的自由度：

  1. Tool 定义要精确——Tool 的 description 写得越清楚，LLM 选对的概率越高
  2. Function Calling 输出 JSON Schema——结构化输出比自由文本可靠 10 倍
  3. 关键路径设置人工确认点——涉及金钱/库存的 Tool 调用必须人工确认
  4. 超时兜底——LLM 卡住了要有超时，不能让 Agent 永远等

总共约 6 周就能达到”能在面试中聊 Agent”的水平。8 周可以做一个小开源项目。

大部分 Agent 开发者是 ML 背景——代码能跑但不懂可靠性。而你的 SyncTask 引擎经验正好是他们的盲区：

  • 他们知道怎么调 LLM，但不知道怎么设计幂等
  • 他们知道怎么写 Prompt，但不知道怎么做告警分级
  • 他们知道怎么调用 Tool，但不知道分页拉取和断点续传

你的基础设施能力在 Agent 领域不是减分项，是稀缺资产。

总结

SyncTask 引擎和 Agent 编排引擎本质上是同一个问题：如何在多个步骤之间协调状态、处理失败、保证可靠性。

区别只是决策者从代码变成了 LLM。但 LLM 带来的新问题（幻觉、不确定性、超时）正好可以用你的可靠性设计经验来解决。

如果你也在考虑转型 AI Agent 开发，建议从自己最熟悉的系统出发，找到对应关系——你会发现大部分经验都可以带过去，需要补的新东西其实不多。
