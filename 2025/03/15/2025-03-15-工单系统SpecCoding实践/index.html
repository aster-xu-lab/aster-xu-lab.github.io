从0到1交付一个中台：工单履约系统的Spec Coding实践

从 0 到 1 交付一个中台：工单履约系统的 Spec Coding 实践

先有 Spec，后有代码

2024 年，公司决定做一个通用的工单履约管理中台，覆盖 3C 维修、贴膜、回收、导购四条业务线。我被指定为技术负责人，从 0 开始设计整个系统。

传统做法是拿到需求就开始建项目——拉 Spring Boot 脚手架、建表、写 CRUD。这次我换了个方式：先写 Spec，再写代码。这个决策让整个交付过程少走了很多弯路。这篇文章记录我的完整实践。

一、什么是 Spec Coding

Spec Coding 的核心思想很简单：

写代码之前，先把”要做什么”写得足够清楚，清楚到可以让 AI 理解并辅助实现。

它不是重文档，而是轻量级的决策记录。一份 Spec 包含四样东西：

  1. 产品分析——业务需求、用户故事、核心流程
  2. 领域建模——实体关系、状态机、限界上下文
  3. API 设计——接口契约（先定契约再写实现）
  4. ADR（Architecture Decision Record）——关键架构决策及理由

二、第一步：产品分析

工单系统的核心流程：

四条业务线（维修、贴膜、回收、导购）的流程 80% 相同，但 20% 不同：

  • 维修需要备件管理
  • 回收需要估价和质检
  • 贴膜最简单，一个技师一种膜
  • 导购需要预约时段

所以不能用四条业务线做四套代码。需要一个通用工单引擎 + 业务线差异配置。

三、第二步：领域建模

核心实体

双状态机协同

这是最核心的设计。工单有工单的状态（待分配→已分配→服务中→待确认→已完成→已关闭），任务有任务的状态（待接收→执行中→已完成→已取消）。

两者需要协同：所有任务完成 → 工单进入”待确认”；工单关闭 → 未完成的任务自动取消。

为什么工单用表驱动而任务用实例化？

  • 工单状态少（6 个），变迁规则简单，表驱动足够
  • 任务类型多（维修任务、备件任务、质检任务…），每类任务的流转规则不同，实例化更灵活

四、第三步：API 设计（契约先行）

在写任何实现代码之前，先定义 API 契约：

先把接口文档给到前端同学 review，确认字段设计合理后再动手写 Controller。这个顺序很重要——一旦 API 定了，前后端就能并行开发。

五、第四步：ADR 架构决策记录

每个关键决策都写 ADR，记录”当时为什么这么选”。

ADR-001: 工单状态机用 COLA StateMachine

决策：使用阿里 COLA 框架的 StateMachine 组件。
理由：(1) 轻量，不依赖 Spring；(2) 支持 UML 风格的状态转换定义；(3) 可视化状态图可以直接生成文档。
备选方案：Spring StateMachine（太重，学习成本高）、自研（没必要重复造轮子）。

ADR-002: 分布式事务用 Seata AT 模式

决策：Seata AT 模式。
理由：工单创建涉及多表写入（工单表 + 任务表 + 通知表），需要保证原子性。AT 模式对业务侵入最小，适合快速上线。
后续计划：压测后如果 AT 性能不达标，切换到 TCC（参考 OFS 订单系统的经验）。

ADR-003: 模板配置使用快照机制

决策：模板本身存储 JSON，下发时生成快照保存到 template_snapshot 表。
理由：模板可能随时被运营修改，但已创建的工单必须绑定创建时的模板内容。快照保证了配置变更不影响历史工单。

六、AI 加速：把 Spec 喂给 AI

Spec 写完后，我把整个 Spec 给 Claude：

  1. 设计评审：让 AI 找出 Spec 中的设计漏洞——它找出了状态机里”待确认→已关闭”的边界条件没处理
  2. 代码生成：根据 API Spec 生成 Controller + Service 框架代码（节省了约 40% 的样板代码时间）
  3. 测试用例：根据状态机定义生成状态迁移测试矩阵

AI 不是替代了我的设计工作，而是加速了设计到代码的转化。

七、成果

指标 数据
项目周期 从 Spec 到提测 3 周
代码规模 182 个 Java 文件，1.2 万行
技术栈 Spring Cloud Alibaba + Dubbo + Seata + COLA
业务线 4 条（维修/贴膜/回收/导购）
Spec 文档 4 份（产品分析/领域模型/API设计/ADR）

最关键的不是代码写得快，而是三个月后新人接手维护时，他能通过 Spec 理解整个系统的设计意图。代码告诉你怎么做，Spec 告诉你为什么这么做。

总结

Spec Coding 的本质：把决策上下文和代码当作同等重要的交付物。

不要把所有时间花在写代码上，花 20% 的时间把”为什么这么设计”写清楚。剩下的 80% 可以让 AI 帮你加速实现——但前提是 Spec 要写得足够清晰，AI 才能理解。
