渠道垂直分表的困局与ShardingSphere破局

渠道垂直分表的困局与 ShardingSphere 破局

从手工拆表说起

订单中心对接 20+ 电商平台（淘宝、京东、拼多多、抖音…），不同渠道的订单量差异巨大。最初的设计是按渠道垂直分表：

这在早期（日均 10-20 万单）是够用的。但随着业务增长，拼多多单渠道日订单突破 50 万，单表超过 2000 万行，查询开始明显变慢。

手工垂直分表走到了尽头。

一、垂直分表的三个困局

困局 1：单表数据量无法控制

垂直拆分只解决了”不同渠道分开存”，但解决不了”单渠道数据量爆炸”。

困局 3：SQL 代码里散落着表名

二、为什么选 ShardingSphere-JDBC？

选型对比：

方案 模式 优点 缺点
ShardingSphere-JDBC JDBC 层拦截 无额外部署，对业务透明 需要引入 jar 包
ShardingSphere-Proxy 独立代理层 对应用完全透明 额外部署运维，多一跳延迟
MyCat 代理层 功能丰富 社区活跃度下降，性能有损耗
Vitess 代理层 YouTube 级验证 MySQL 兼容性有限，运维复杂
自研路由 代码层 最灵活 维护成本高，团队负担大

对于订单中心这种内部 B 端系统，JDBC 模式是最合适的：

  • 不需要额外部署服务（Proxy 需要高可用部署）
  • 对业务代码改动最小（改 SQL 注解即可）
  • 性能损耗最小（JDBC 层直接改写 SQL，没有网络跳转）

三、改造方案

按订单号哈希分片

效果：2 库 × 4 表 = 8 个分片，单表数据量从 2300 万降到 ~300 万。

分片键的选择

选择 order_no（订单号）作为分片键的原因：

  1. 唯一性：每个分片键值对应唯一一条订单
  2. 查询模式：99% 的查询都带订单号
  3. 分布均匀：订单号是随机生成的，哈希后均匀分布

不选 user_id 的原因：用户维度的查询很少，不值得为了低频查询牺牲高频查询性能。

四、分库分表的副作用和解决方案

副作用 1：跨分片 JOIN 失效

解法：order_item 也用 order_no 做分片键，保证同一个订单的所有数据落在同一个分片。

副作用 2：深度分页

解法：禁止深度分页，改用”游标分页”或 ES 搜索。

副作用 3：分布式事务

跨分片的写入需要分布式事务保证，用 Seata AT 模式：单分片内是本地事务，跨分片走全局事务。

副作用 4：全局 ID

分片环境下自增 ID 会冲突，改用百度 UidGenerator（雪花算法变体，性能更好）。

五、渐进式迁移策略

直接在核心订单表上做分库分表风险太大，我们采用渐进式迁移：

  1. 第一步（试点）：选一个量最大的渠道（拼多多），新建 order_pdd_sharded 表组（2库4表），双写验证一个月
  2. 第二步（灰度）：切 10% 流量到新表，观察一周
  3. 第三步（全量）：切 100% 流量，老表保留 30 天作为回滚备份
  4. 第四步（推广）：验证稳定后，推广到其他渠道

六、下一步：CDC 实时同步

分库分表解决的是单表容量问题，但带来了一个新需求：数据分析（Doris/ClickHouse）需要从多个分片汇总数据。

方案是引入 CDC（Change Data Capture）：

MySQL 的写操作通过 binlog 实时同步到 Doris，OLAP 查询走 Doris 而不是直接查 MySQL 分片。这样写入走分库分表，分析走 Doris，读写分离。

总结

分库分表不是银弹，是数据库容量问题的最后手段。在做之前先问自己：

  1. 索引优化做完了吗？
  2. 冷热数据分离了吗？
  3. 读写分离了吗？
  4. 能用缓存扛吗？

如果以上都做了还不够，再考虑分库分表。核心是理解副作用并做好应对：

  • 分片键决定查询性能的 90%
  • 跨分片 JOIN 要通过冗余字段或绑定分片键解决
  • 深度分页直接禁止，换游标或 ES
  • 渐进式迁移，不要一步到位
