放弃DAG：为什么我用afterExecute埋点替代了编排引擎

放弃 DAG：为什么我用 afterExecute 埋点替代了编排引擎

一个有争议的架构决策

同步任务引擎中有一个高频场景：订单同步完成后需要自动触发库存同步，库存同步完成后触发物流同步。这是一个典型的链式依赖。

遇到链式依赖，99% 的工程师第一反应是：”用 DAG 编排引擎啊！”

我最初也是这样想的。我调研了 Temporal、Conductor、nFlow、Flowable、Airflow、Azkaban 六个方案，然后做了一个让很多人意外的决定：全部放弃，用最简单的 afterExecute 埋点实现。

这个决策在当时是有争议的。今天这篇文章记录完整的思考过程。

一、问题定义：我们到底要编排什么？

先看同步场景：

这种依赖有两个特点：

  1. 链是静态的——不需要运行时动态决定下一步走哪个分支
  2. 中间态需要持久化——每一步完成后，下一条可能 30 秒后才开始执行

二、六个方案的调研

方案 类型 强项 致命问题
Temporal Workflow-as-Code 最强大，支持 Saga、重试、超时 太重，需要独立 Server
Netflix Conductor JSON DSL 编排 可视化，社区好 需要独立部署 + 学习 JSON DSL
nFlow 阿里开源 轻量 社区活跃度低，文档少
Flowable BPMN 引擎 标准 BPMN 2.0 太重，面向审批流不是数据同步
Airflow Python DAG 调度强 Python 生态，Java 集成成本高
Azkaban Hadoop 生态 批处理调度 跟 Java 微服务不搭

六个方案共同的负担：

  • 都需要独立部署和维护
  • 都需要团队学习新的 DSL 或 API
  • 都要管理额外的运行时状态

三、转折点：一个简单的问题

我问了自己一个问题：

我们的同步链只有 11 个 Handler，链的结构半年才变一次。引入一个新基础设施，真的值得吗？

算一笔维护账：

  • 引入 Temporal：部署 Temporal Server + 学习 Workflow SDK + 排查分布式问题 + 版本升级
  • 自己实现 Choreography：在现有 sync_task 表上加一个 context.properties 字段就够了

投入产出比完全不成比例。

四、afterExecute 埋点：Choreography 模式

Choreography 的核心思想：没有中央指挥，每个参与者做完自己的事，留一个信号，下一个参与者自动响应。

五、Orchestration vs Choreography

维度 Orchestration (DAG) Choreography (afterExecute)
控制流 中央编排器指挥 参与者自发响应
可追踪性 一目了然（DAG图） 需要日志拼接
耦合度 编排器知道所有步骤 每个步骤只知道下一跳
灵活性 改流程改DAG配置 改 afterExecute 代码
维护成本 运行+维护编排引擎 零额外基础设施
适用规模 复杂分支流程 简单链式流程
新人理解 需要学 DSL 直接看 afterExecute

六、什么时候不该用 Choreography？

这个决策不是普适的。如果你的场景满足以下任一条件，应该用 DAG：

  1. 流程有复杂的分支和合并（菱形依赖、并行汇聚）
  2. 流程频繁变化（运营人员需要可视化编排）
  3. 步骤数 > 30（afterExecute 散落在各 Handler 中难以追踪）
  4. 需要流程级别的超时和补偿（不是单个步骤的超时）

我们的场景（11 个 Handler、静态链式、半年一改）恰好是 Choreography 的甜点区。

七、面试话术：如何讲好这个决策

如果面试官问”你们怎么不做 DAG”，可以这样回答：

我评估了 Temporal、Conductor、nFlow 等 6 个方案，核心判断是：场景不匹配。我们的同步链只有 11 个 Handler，链的结构半年不变。引入 DAG 编排引擎相当于为 11 个步骤维护一个额外的分布式系统，投入产出比不成比例。

我选择了 Choreography 模式——afterExecute 埋点 + sync_task 表的 context 字段作为事件载体。每个 Handler 完成后在 context 里标记信号，下一个 Handler 自发响应。

零引擎、零 DSL、零维护成本。这不是说 DAG 不好——如果步骤数到 30+ 或者需要可视化编排，我会选 Temporal。做架构就是选最适合当前场景的方案，而不是选最酷的方案。

总结

放弃 DAG 不是放弃编排，而是选择了更轻量的编排模式。架构决策的核心不是”哪个方案更强”，而是”哪个方案最匹配当前场景”。

如果你也在做类似的技术选型，建议先问自己三个问题：

  1. 流程复杂度：有多少个步骤？分支多吗？
  2. 变化频率：流程多久改一次？谁改？
  3. 维护成本：团队有精力维护编排引擎吗？

回答完这三个问题，你的选择应该就清楚了。
